# Revision v1.9 — 2026-07-23 — Non-goals vs. deferred: split the out-of-scope list (docs-only Addendum)

Status: Adopted 2026-07-23. Docs-only: this revision changes
`docs/PROJECT_PLAN.md` and adds this file. No code, no models, no
vars, no thresholds, no schedules.

## Rationale

Section 3's single "Explicitly Out of Scope" list conflated two
different kinds of exclusion and read as prohibiting both forever:

- **Thesis-level non-goals** — things this project *is not* (a Strava
  clone, a prediction engine, a mobile app). "Machine-learning
  models, performance predictions, race-time predictions" is not a
  deferral; it conflicts with the project's observational-never-causal
  stance.
- **MVP deferrals** — things excluded to bound the initial build, not
  rejected ("Real-time processing", "Complex cloud deployment").

The webhook/FastAPI extension surfaced the ambiguity: an event-driven
ingestion path reads as flatly prohibited under "Real-time processing"
and "Complex cloud deployment", when those lines were build-bounding
deferrals, not statements of purpose. Precedent already exists for
deferred work entering scope by addendum — cardiac drift (Release
1.1) and Airflow (v1.5) — but the flat list gave that precedent no
home.

This is a governance clarification, not a loosening. The
decision-addendum process stays exactly as it is.

## What changes

1. Section 3's "Explicitly Out of Scope" splits into two subsections:
   **"Non-Goals (this project's thesis)"** (preamble: revising the
   list means revisiting the project's purpose, not filing a routine
   addendum) and **"Deferred (out of scope for Release 1.0)"**
   (preamble: any deferred item may enter scope through the standard
   decision-addendum process, at which point it moves from the list
   into that addendum's scope statement).
2. The Decision Log preamble gains one sentence: out-of-scope items
   are release-scoped unless listed as non-goals; non-goals are
   statements of the project's purpose, revised only by revisiting
   that purpose; entering a deferred item requires an addendum, never
   a plan rewrite.

## Item mapping (old list → new lists)

Every item of the old list appears in exactly one new list; nothing
was added or dropped. Two items are rephrased to name what was
actually being excluded; no rephrasing changes what is excluded.

| Old line | New list | New line |
|---|---|---|
| Replication of standard Strava screens | Non-Goals | unchanged |
| Machine-learning models, performance predictions, race-time predictions | Non-Goals | "Machine-learning performance or race-time prediction", with the observational-never-causal rationale stated inline |
| Mobile application development | Non-Goals | unchanged |
| Real-time processing | Deferred | "Real-time / event-driven ingestion (e.g., Strava webhooks)" |
| Complex cloud deployment | Deferred | "Cloud-hosted components (e.g., an always-on webhook receiver)" |

## What does NOT change

- **No locked decision changes.** D1–D22 are untouched; this addendum
  amends no decision.
- **No scope enters or leaves.** Items are re-categorized only. The
  webhook/FastAPI extension is NOT hereby admitted — entering scope
  still requires its own addendum with its own scope statement.
- **The decision-addendum process is unchanged.** This revision names
  the existing path (deferred item → addendum → scope statement); it
  does not create a new one.
- Every other in-scope/out-of-scope line in section 3 stays where it
  is.
