# Revision v1.5 — 2026-07-16 — Airflow as a thin scheduling and observability layer (Addendum to D18)

Status: Adopted 2026-07-16. Docs-before-code: this addendum lands
before the implementation session — no DAG code and no installation in
the revision's docs commit. First revision recorded under
`docs/decisions/` rather than appended to `docs/PROJECT_PLAN.md`; the
plan carries a pointer stub at the end of its revision ledger.

## Rationale

D18 locked orchestration as Makefile + Python CLI, with an explicit
escape hatch: "No Airflow in v1 — added later only if it improves the
actual workflow, not as a résumé keyword." That later is now. The MVP
and every release through v1.4 are delivered, and the recurring
workflow is a real multi-step chain — activities → coordinate
backfill → weather → streams → dbt build/test — currently run by
hand, with the stream backfill deliberately draining across successive
`make sync-streams` invocations. What the workflow lacks is scheduling
and run observability (when things ran, what failed, where to look),
not execution logic: the pipeline already owns idempotency,
resumability, and all ingestion state.

Airflow is therefore adopted post-MVP as a THIN scheduling and
observability layer. It decides *when* to run and shows *what
happened*; the pipeline decides *what to do*. D18's division of labor
is kept, not replaced.

## Revised decision

**D18 (revised v1.5):** Orchestration remains Makefile + Python CLI as
the sole execution layer. Airflow schedules and observes it, under
five constraints:

- (a) Airflow owns no state — watermarks, per-item status rows, and
  destination-as-cache remain the pipeline's;
- (b) catchup=False and no templated date windows, because Strava
  filters by activity start date and interval-based windows would
  reintroduce the late-upload gap the 14-day overlap already solves;
- (c) tasks invoke existing Make targets unchanged — no pipeline code
  changes;
- (d) LocalExecutor-or-simpler, SQLite metadata DB, no
  Celery/Redis/Docker for Airflow in this release;
- (e) Airflow lives in its own venv, never in the project's
  dependencies.

## Why each constraint holds

- **(a) State.** The sync watermark lives in `raw_strava.sync_state`
  and advances only after a fully successful sync; stream fetches
  carry per-row terminal statuses in `raw_strava.streams`
  (success/unavailable terminal, failed retries next run); weather has
  no watermark at all — `raw_weather.hourly` IS the cache. These are
  the pipeline's idempotency and resume contracts, tested where they
  live. Airflow's run bookkeeping must never become a second,
  competing source of truth about what has been ingested.
- **(b) Scheduling model.** Strava's API filters by *activity start
  date*, not upload date. A run recorded Monday but uploaded Thursday
  falls outside any Monday-shaped interval window — exactly the gap
  D6's 14-day overlap closes. Templated
  `data_interval_start`/`data_interval_end` windows would re-open it.
  Every scheduled run is therefore an ordinary incremental sync from
  the pipeline's own watermark, and `catchup=False` because a missed
  scheduler interval needs no replay: the next run's overlap covers
  it.
- **(c) Execution surface.** A task runs the same `make` target an
  operator would type, so scheduled and manual runs are the same code
  path. Behavior reachable only through Airflow would sit outside
  `make test`'s coverage; none is permitted.
- **(d) Deployment weight.** One machine, one athlete, no concurrent
  heavy workloads. LocalExecutor-or-simpler on a SQLite metadata DB
  matches that scale; a Celery/Redis/Docker stack for Airflow would
  invert the thin-layer premise.
- **(e) Dependency isolation.** The project venv is pinned (Python
  3.13, dbt-core compatibility); Airflow's large, tightly-pinned
  dependency tree must neither constrain nor be constrained by it.
  Structural consequence: the project never imports Airflow, so
  `make test` and `make lint` pass with Airflow absent.

## What this revision does not change

- The Make targets and CLI stay the canonical, fully supported way to
  run everything by hand; `make all` works with Airflow stopped or
  never installed.
- No pipeline code, dbt model, threshold, schema, or test changes; no
  new D-number — this revises D18 only.
- All other decisions (including D19's dashboard rules and D21's MVP
  boundary) are untouched.

## Acceptance criteria (for the implementation session)

1. Every Airflow task is a thin invocation of an existing Make
   target; no task contains pipeline logic, SQL, or credentials.
2. `make test` and `make lint` pass in the project venv with Airflow
   not installed — proving constraint (e) structurally.
3. Constraint (b) is grep-provable in DAG code: `catchup=False` on
   every DAG; no `data_interval`, `execution_date`, or other templated
   date-window usage.
4. A scheduled run and a manual `make all` are indistinguishable from
   the pipeline's side: same watermark advance, same per-item status
   rows, same cache writes.
5. Airflow runs LocalExecutor or simpler on a SQLite metadata DB, from
   its own venv; no Celery, Redis, or Docker services are added for
   Airflow.

## Exit-code-3 policy (2026-07-16, decided with the DAG plan)

The pipeline CLI exits **3** for a clean rate-limit/budget stop:
committed work is kept, the activities watermark is not advanced, and
every sync is resumable (all four sync subcommands carry the
contract). In the DAG this maps to task state **SKIPPED** via
BashOperator `skip_on_exit_code=3` — visibly amber, never red, never
falsely green — and every downstream task carries
`trigger_rule="none_failed"`, so `dbt_build` still runs on
already-ingested data after an early stop while a genuine failure
(exit 1) halts the chain.

**Narrow amendment to constraint (c), chosen here, not absorbed:**
GNU make replaces any failing recipe's exit status with its own exit 2
(verified empirically), so exit 3 cannot survive `make <target>` and a
clean stop would be indistinguishable from failure. The four sync
tasks therefore invoke the CLI directly —
`.venv/bin/running-pipeline <cmd>`, the verbatim one-line body of the
corresponding make recipe, run with cwd = repo root — rather than
`make`. Constraint (c)'s intent is intact: no pipeline logic in DAGs,
and scheduled runs share the manual code path exactly. `dbt_build`,
the one recipe with real logic (profile bootstrap, `.env` sourcing)
and no exit-3 contract, stays behind `make dbt-build`, where make's
exit-2 masking is harmless because any nonzero is genuine failure.
