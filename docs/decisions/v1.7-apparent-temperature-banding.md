# Revision v1.7 — 2026-07-22 — Band runs by apparent temperature; drop the weekly trend band (Addendum to D14)

Status: Adopted 2026-07-22. Docs-before-code: this addendum lands as a
docs-only commit; the interface change, the banding switch, and the
dashboard wording follow in separate commits on the same branch.

## Rationale

The efficiency-by-temperature analysis wants the exposure a runner
actually experiences. Open-Meteo's apparent temperature folds humidity
and wind into a single feels-like value that is already fetched
(D8 hourly variables), staged (`apparent_temperature_f`,
`stg_weather__hourly`), and carried through to `fct_runs` — so
switching the D14 band predicate's input incorporates humidity into
the weather-band analysis with no new API fields, no new raw or
staging columns, and no new bands.

Separately, `mart_efficiency_trend`'s weekly-average band is unsound
at its own grain: the band of a weekly average can describe no run
that happened (a 40°F run and an 80°F run band as "mild"), and at the
D12 floor of 2 runs the average is fragile near band boundaries.
Run-level banding in `mart_efficiency_by_temp_band` is the sanctioned
weather-adjustment mechanism — its dashboard caption already states
that runs are banded individually, never averaged by week. So the
weekly band columns are removed rather than migrated. The week's
`avg_temperature_f` stays as unbanded context and joins the weekly
trend chart's tooltip (beside the existing avg-HR line), which is
where the confounder context actually gets seen.

## What changes

1. **D14 banding input (Change A).** The two run-level call sites of
   the `temperature_band_range` macro switch their input from
   `runs.temperature_f` to `runs.apparent_temperature_f`:
   `mart_efficiency_by_temp_band` and `mart_run_quality`. The macro
   and the seed bounds are untouched — only the expression passed at
   the call sites changes.

   **Change A is NOT output-invariant: band assignments are expected
   to move for any run whose dry-bulb and apparent temperatures
   straddle a seed boundary.** On the 2026-07-22 warehouse the
   migration preview happens to be empty (14 weather-matched valid
   outdoor runs: mild→mild 3, warm→warm 11, zero movers) — the
   band-migration report produced at verification time, not the
   accident of zero movers, is the proof of behavior.

2. **The weekly trend band is removed (Change B).**
   `mart_efficiency_trend` drops `temperature_band_key` and
   `temperature_band_label` along with its `temperature_bands` seed
   join — the third macro call site is removed, not migrated, leaving
   exactly the two run-level call sites above. Their marts.yml column
   docs go with them. `avg_temperature_f` stays (dry-bulb weekly
   average; context only).

   **Change B is an interface change under the v1.3 rule — column
   set and column semantics are a contract, and this reshape is
   chosen and recorded here, not absorbed from a diff.**

3. **The band ladder becomes a partition (coverage rule).**
   `weather_available` is timing-only (a matched observation needs
   any one of temperature / apparent / humidity / wind), so a valid
   outdoor run can be weather-available yet carry no apparent
   temperature. Today such a run would silently vanish from
   `mart_efficiency_by_temp_band` — matching neither a seed band nor
   `indoor` nor `no_weather` — which violates data-quality
   principle 1 (nothing is silently filtered). **Fallback to
   `temperature_f` is not allowed.** Instead `no_weather`'s predicate
   widens to `not is_trainer and (not weather_available or
   apparent_temperature_f is null)`: the pseudo-band now means "no
   matched feels-like temperature". `mart_run_quality` needs no SQL
   change — its coalesce already labels such runs
   'weather unavailable'; this revision makes that label
   correct-by-definition. Reachability, recorded pre-implementation
   (2026-07-22): 720 measurement-carrying staged observations, zero
   missing either temperature; 20 valid runs, zero in the gap on
   either the current or the post-switch variant — so the red proof
   runs on scratch-database fixtures, not the warehouse.

4. **Trainer guard (chosen scope extension, signed off).**
   `mart_efficiency_by_temp_band`'s seed-band join lacked the
   `not runs.is_trainer` guard that `mart_run_quality`'s join has, so
   a trainer run with matched weather would count in both `indoor`
   and a seed band. Zero real rows hit this today, making the fix
   output-invariant on current data; it lands in the same commit that
   makes the ladder a partition, proven red first by a
   trainer-with-weather fixture run.

5. **Feels-like context columns.** `mart_efficiency_by_temp_band`
   gains `avg_apparent_temperature_f` (the average feels-like
   temperature of the band's runs; NULL for `indoor` and
   `no_weather`). `mart_run_quality` gains per-run
   `apparent_temperature_f`, displayed as "Feels-like °F" beside the
   dry-bulb column, whose header is renamed "°F" → "Air °F" so the
   pair explains itself without a caption.

## What does NOT change

- **Seed bounds, labels, band count, sort order.** The
  `temperature_bands` seed is byte-identical; both staged
  temperatures round to 1 dp in staging, so the 49.9 / 50.0 / 70.0 /
  70.1 bound alignment holds for apparent exactly as it did for
  dry-bulb.
- **Seed column and macro names.** `min_temperature_f` /
  `max_temperature_f` and `temperature_band_range` keep their names —
  the bounds are still degrees Fahrenheit; only what they now bound
  (apparent temperature) is re-documented. Renaming would be pure
  churn.
- **Every existing `avg_temperature_f` column stays dry-bulb.** No
  column is silently re-meant (v1.3 rule). `mart_band_weekly`'s
  "context only (dry-bulb)" note remains true, as does every other
  context copy. In `mart_efficiency_by_temp_band` the column is now
  the dry-bulb average of an apparent-assigned band — documented as
  such — and the `no_weather` row keeps its contractual NULL even
  though that row can now hold runs that do carry dry-bulb readings.
- **`avg_apparent_temperature_f` is deliberately mart-only.** It
  exists for auditability (a band's assignment is checkable from the
  mart itself) and is intentionally NOT surfaced in the app. A future
  session must not "fix" the asymmetry by surfacing it — that would
  be scope drift in the other direction.
- **The band_key vocabulary** (`cold` / `mild` / `warm` / `indoor` /
  `no_weather`), its accepted_values test, and the app's ordinal
  color ramp are untouched.
- **Out of scope**: dew point (possible separate future revision);
  band bounds, band count, or new thresholds of any kind; the
  hr_bands / D22 band models; `weather_client.py`, the raw_weather
  schema, and the staging column set; any dashboard change beyond the
  band wording, the tooltip line, and the sanctioned Air/Feels-like
  column pair; the D19 allow-list (no mart is added or removed).

## Consistency contract

A new run-grain singular test
(`assert_temp_band_ladder_assigns_every_valid_run_once`) proves the
ladder is a partition: every valid run lands in exactly one of
{seed band, indoor, no_weather}. It mirrors
`mart_efficiency_by_temp_band`'s ladder leg-for-leg and must be
edited in lockstep with it. It exists alongside — not instead of —
the aggregate conservation test
(`assert_band_mart_accounts_for_every_valid_run`), because an
aggregate sum can be fooled by offsetting errors (one dropped run
plus one double-counted run) that a run-grain check cannot. Both
failure arms are proven red first (a matched-weather run with NULL
apparent temperature → zero assignments; a trainer run with matched
weather → two assignments), and a permanent pytest re-proof injects
an unassignable run after a green build and asserts the test catches
it — the v1.3 orphan-injection pattern.

## Dashboard (D19: clarified, not amended)

The three-view cap and the pinned allow-list are untouched — no mart
is added or removed; only columns within existing marts change. The
weekly trend chart's point tooltip gains one line, avg temp (°F),
after the existing avg-HR line. The run-quality table shows the
sanctioned "Air °F" / "Feels-like °F" pair. The
efficiency-by-temperature-band caption changes to say runs are banded
by their own matched apparent temperature (feels-like); the "Week
temp band" table column disappears with Change B. Nothing else in the
app changes.
