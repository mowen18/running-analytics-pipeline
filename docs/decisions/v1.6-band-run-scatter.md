# Revision v1.6 — 2026-07-20 — Run×band scatter mart for the band view (Addendum)

Status: Adopted 2026-07-20. Docs-before-code: this addendum lands as a
docs-only commit; the mart, its contracts, and the chart redesign
follow in separate commits on the same branch.

## Rationale

The pace-at-HR-band chart plots week × band medians. At the current
data volume (typically 1–3 contributing runs per week per band), a
weekly median-of-run-medians point nearly coincides with the run-level
medians it summarizes — the chart draws the same information twice
while hiding the run-level spread that would show how tight or noisy a
band's week actually was. The run-level values already exist at
run × band grain in core (`fct_run_band_segments`, since v1.4); the
dashboard just has no sanctioned mart to read them from (D19: marts
only).

## What changes

One new mart, and a redesign of the band chart that reads it.

| Layer | Model | Grain | Key contents |
|---|---|---|---|
| Mart | `mart_run_band_segments` | run × band | activity_id, start_date_local, week_start_date, band_key / band_label / band_sort_order, median_velocity_m_per_s, median_pace_min_per_mi (the run's band median), dwell_min |

- The mart is an explicit-column projection of core
  `fct_run_band_segments` joined to the `hr_bands` seed for display
  columns (label, sort order) — the `mart_run_drift` pattern at
  run × band grain. No filters: every analyzed run × band row.
  `median_velocity_m_per_s` is carried for the consistency contract
  below; `dwell_min = round(dwell_s / 60.0, 1)` follows
  `analysis_window_min`'s unit-conversion pattern.
- The chart becomes one faint point per run per band (x = run date,
  y = the run's band median pace) with the 28-day rolling median line
  per band as the ONLY line, still connecting sufficient weeks only.
  Weekly-median points leave the chart: they duplicated the run points
  at this volume. Weekly sufficiency remains visible as line
  connectivity and as the `is_sufficient` flag in the table; the
  weekly medians themselves remain in the table (`mart_band_trend`),
  unchanged.

## What does NOT change

- **D22's weekly statistic is unchanged.** `mart_band_weekly` still
  computes the median across contributing runs of run-level band
  medians (D11's median-of-runs philosophy), and `mart_band_trend`
  still carries it with the D12 sufficiency flag.
- **`mart_band_weekly`'s output is byte-identical.** The new mart is a
  pure leaf — nothing upstream of any existing mart moves — so the
  v1.4 bidirectional-EXCEPT invariance check is not triggered. The one
  edit to `mart_band_weekly.sql` in this revision is comment-only (no
  executable SQL changes).
- **No threshold changes**: `min_weekly_valid_runs` (2),
  `band_min_dwell_minutes` (5), `stream_fetch_min_moving_minutes`
  (20), and every other var stay exactly as they are.
- **No layer-matrix change.** Marts may already read core, seeds, and
  marts; the new mart reads core `fct_run_band_segments` plus the
  `hr_bands` seed. (Contrast v1.4, which needed one deliberate
  widening; v1.6 needs none.)

## Consistency contract

A singular test (`assert_band_weekly_matches_segment_mart`) pins the
two grains together: for every week × band,
`mart_band_weekly.median_velocity_m_per_s` must equal
`round(percentile_cont(0.5) within group (order by the new mart's
per-run median_velocity_m_per_s), 3)`, `contributing_run_count` must
equal the mart's row count, and both sides must cover exactly the same
week × band keys. The comparison lives in VELOCITY space only, for two
reasons: percentile interpolation does not commute with the 1/x pace
transform (two runs at 2 and 4 m/s: weekly median velocity 3 m/s
→ 8.94 min/mi, but the median of the two run paces is 10.06), and the
weekly pace column is derived from the unrounded weekly percentile, so
even odd-count pace comparisons could differ in the last decimal
place. The row-count check exists because a seed-join fan-out would
leave a uniformly duplicated multiset's median unchanged — median
equality alone cannot catch it.

## Dashboard (D19: clarified, not amended)

The band section stays inside the Aerobic Efficiency view; the
three-view cap holds. The app's pinned allow-list grows by exactly one
name (`mart_run_band_segments`), proven red first with both negative
cases intact — the v1.4 acceptance-criterion-7 pattern. Run points
carry run date, band, the run's band median pace, and minutes in band
in tooltips; the multiselect band filter and the reversed pace axis
(up = faster) carry over unchanged.
